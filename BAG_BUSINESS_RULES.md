# 背包业务规则说明

这份文档只讲业务规则，不讲接入步骤。  
如果你要知道怎么调用，请看 `README.md`；如果你要知道“什么情况下允许、什么情况下拒绝”，看这里。

## 1. 规则目标

Bag 插件的规则目标是：

- 背包数据可落库、可追踪、可回放。
- 高危操作必须由服务端决策。
- 同一玩家的写操作保持顺序一致。
- 客户端只能触发整理类动作，不能直接绕过校验。
- 规则尽量简单，后续可扩展，不把业务逻辑锁死在存储层。

## 2. 核心概念

### 2.1 `BagItem`

一条 `BagItem` 代表一个物品实例，不是“某种物品的总数”。

关键字段：

- `bid`：实例唯一 ID。
- `itemID`：配置 ID。
- `count`：当前数量。
- `bagIndex`：格子位置。
- `stackable`：是否允许堆叠。
- `maxStack`：单格最大堆叠数。
- `bindType`：绑定类型。
- `expireAt`：过期时间戳。
- `locked`：锁定状态。
- `extra`：附加属性。

### 2.2 `capacity`

`capacity` 表示背包容量，单位是“可容纳的实例条数”。

- `0` 表示不限制容量。
- `>0` 表示最多允许这么多条 `BagItem` 存在。

注意：当前容量不是重量、体积，也不是总堆叠数。

### 2.3 `opID`

`opID` 是业务操作号。

建议用途：

- 奖励发放追踪
- 商城补单追踪
- 邮件附件追踪
- 交易流水追踪
- GM 工具追踪

`opID` 不是强制，但强烈建议填写。

## 3. 容量规则

### 3.1 什么时候检查容量

以下操作需要检查容量：

- `addItem`
- `splitItem`
- `transferItems`

### 3.2 容量判定

- `capacity = 0` 时，不做容量限制。
- `capacity > 0` 时，新增实例前必须确认还有空位。
- 只要会产生新实例，就要算容量。

### 3.3 容量与堆叠的关系

容量看的是“实例条数”。

例如：

- 一个 99 堆叠的药水，只占 1 个格。
- 但如果继续加满，拆成 2 格，就会消耗 2 个容量。

## 4. 堆叠规则

### 4.1 堆叠前提

只有同时满足以下条件，才允许叠到已有实例上：

- `stackable = 1`
- `itemID` 相同
- `extra` 相同
- `locked` 不阻止该业务操作
- 当前数量还未达到 `maxStack`

如果 `itemID` 或 `extra` 有任意一个不同，就不能合并到同一实例里，必须保持为两个独立实例。

### 4.2 `addItem`

`addItem()` 的执行顺序：

1. 优先找同 `itemID`、同 `extra`、允许堆叠、未满的实例。
2. 能叠多少就先叠多少。
3. 剩余数量再拆到新实例。
4. 新实例位置放到末尾。

### 4.3 `maxStack`

`maxStack` 是单个实例允许的最大数量。

- 大于 1：表示可堆叠。
- 等于 1：等价于单件物品。

### 4.4 `splitItem`

拆分规则：

- 源物品数量必须大于拆分数量。
- 拆分后源物品至少保留 1 个。
- 新物品复制源物品的 `itemID`、`stackable`、`extra`、`maxStack`、`bindType`、`expireAt`、`locked`。
- 新物品放到末尾。

## 5. 位置规则

### 5.1 `bagIndex`

`bagIndex` 表示背包格子序号。

规则：

- 小的在前。
- `sortItems()` 会按规则重排。
- 新插入物品默认放在末尾。

### 5.2 `moveItem`

移动规则：

- 目标格为空：直接移动。
- 目标格有物品：交换。
- 目标位置等于原位置：视为无变化。

### 5.3 `swapItem`

交换规则：

- 两个 `bid` 必须不同。
- 交换的是 `bagIndex`，不是物品内容。

### 5.4 `sortItems`

整理规则：

- 按 `itemID` 升序
- 再按 `bid` 升序
- 重新写入连续 `bagIndex`

## 6. 删除与扣减规则

### 6.1 `removeItem`

规则：

- 按 `bid` 扣数量。
- 不能扣成负数。
- 扣到 `0` 时删除这条实例。

### 6.2 `clear`

清空规则：

- 删除该玩家背包全部实例。
- 这是高危操作，只能由服务端调用。

## 7. 合并规则

`mergeItem(fromBID, toBID)` 的判定条件：

- `fromBID != toBID`
- 两个实例都存在
- `itemID` 相同
- `extra` 相同
- `stackable = 1`
- 合并后数量不能超过 `maxStack`

如果 `itemID` 或 `extra` 不一致，就不能合并，应该保留为两个实例。

合并成功后：

- `toBID` 增加数量
- `fromBID` 删除

## 8. 交易规则

### 8.1 `transferItems`

`transferItems(targetDBID, items)` 支持一次交易多个物品。

建议规则：

- `items` 按顺序执行。
- 每一项都要校验源物品存在、数量足够、未被业务锁定。
- 目标背包容量必须足够。
- 对于整件转移，直接转移实例所有权。
- 对于部分转移，源实例扣减，目标实例新增。

### 8.2 当前实现的边界

当前实现是“顺序 SQL”模式，不是同连接事务版。

所以业务上要记住：

- 适合内部工具和普通背包流转。
- 需要强一致的高价值交易，建议后续改成事务版。

## 9. 绑定、过期、锁定

### 9.1 `bindType`

`bindType` 目前是业务保留字段，建议用于：

- 未绑定
- 角色绑定
- 账号绑定
- 其他自定义绑定类型

### 9.2 `expireAt`

`expireAt` 是过期时间戳。

- `0` 表示不过期。
- 到期后是否自动删除、变灰、失效，由业务层决定。

### 9.3 `locked`

`locked = 1` 时，建议业务层拒绝以下操作：

- 删除
- 交易
- 丢弃
- 合并

是否允许整理和移动，可以按项目规则放宽。

## 10. 回调规则

### 10.1 客户端回调

客户端回调只负责通知，不负责决策。

- `onBagList`
- `onBagUpdated`
- `onBagPage`
- `onBagError`

### 10.2 回调开关

每个回调都有独立开关：

- `notifyBagList`
- `notifyBagUpdated`
- `notifyBagPage`
- `notifyBagError`

规则：

- 开关只影响通知，不影响实际写库。
- 关闭 `onBagUpdated` 不代表操作不执行。
- 关闭 `onBagError` 只是屏蔽通知，错误仍然会写日志。

## 11. 日志规则

### 11.1 什么时候写日志

只要操作成功修改了背包，就会写操作日志。所有 9 种写操作均自动记录：

- 添加（ADD）
- 删除（REMOVE）
- 拆分（SPLIT）
- 交换（SWAP）
- 移动（MOVE）
- 合并（MERGE）
- 整理（SORT）
- 清空（CLEAR）
- 交易（TRANSFER）

### 11.2 日志分级

日志支持三级过滤，方便在不同环境（开发/测试/生产）控制日志量。

| 等级 | 常量 | 记录范围 |
| ---- | ---- | -------- |
| L1 | `BAG_LOG_LEVEL_L1` (1) | ADD、REMOVE、CLEAR、TRANSFER |
| L2 | `BAG_LOG_LEVEL_L2` (2) | L1 + SPLIT、MERGE |
| L3 | `BAG_LOG_LEVEL_L3` (3) | 全部操作（L2 + MOVE、SWAP、SORT） |

**适用场景：**

- **L1** — 只追踪资产增减和跨玩家交易，适合生产环境审计。
- **L2** — 额外追踪拆分/合并，适合排查堆叠和整理问题。
- **L3** — 记录所有操作（含拖拽、整理），适合开发调试和压测验证。

默认等级为 L3。低于当前等级的操作类型不会被记录。

### 11.3 输出类型

| 类型 | 常量 | 说明 |
| ---- | ---- | ---- |
| 数据库 | `BAG_LOG_OUTPUT_DATABASE` (1) | 写入 `kbe_plugin_bag_op_logs` 表，支持 SQL 审计查询 |
| 文件 | `BAG_LOG_OUTPUT_FILE` (2) | 写入本地日志文件，自动按日期和大小滚动分割 |

默认输出到数据库。两种输出可以随时切换，无需重启。

### 11.4 文件日志滚动规则

文件日志使用自定义 `DailySizeRotatingFileHandler`，同时按日期和文件大小双重滚动：

- **日期滚动**：跨天自动生成 `bag.2026-05-29.log` 格式的新文件。
- **大小滚动**：同一天内文件超过 `maxBytes` 时，追加编号 `bag.2026-05-29.1.log`。
- **清理策略**：历史文件超过 `backupCount` 时，按修改时间删除最旧的文件。
- **日志格式**：行业标准 key=value 管道格式：

```text
2026-05-29 14:30:15 | INFO | bag-op|level=L1|opType=ADD|ownerDBID=10001|targetDBID=0|bid=42|...
```

### 11.5 配置方法

所有参数通过 Python API 设置，无需编辑配置文件。可以在插件入口、业务脚本或任何地方调用。

```python
from plugins.Bag.common.bag_service import (
    setBagLogConfig,
    setBagLogLevel,
    setBagLogOutputType,
    setBagLogFileConfig,
    getBagLogConfig,
)

# 完整配置（所有参数可选，未传项保持原值）
setBagLogConfig(
    logLevel=2,                          # L2 分级
    outputType=2,                        # 输出到文件
    filePath="logs/bag/bag.log",
    maxBytes=10 * 1024 * 1024,          # 10 MB 滚动
    backupCount=30,                      # 最多保留 30 个历史文件
    encoding="utf-8",
)

# 快捷方法
setBagLogLevel(1)                       # 只改分级
setBagLogOutputType(1)                  # 切回数据库
setBagLogFileConfig(maxBytes=5*1024*1024, backupCount=10)  # 只改文件参数
config = getBagLogConfig()              # 读取当前配置
```

### 11.6 日志原则

- 日志是追踪工具，不是业务真相本身。
- 日志写失败不会把已经成功的背包操作回滚成失败，只会打 `ERROR`。
- 高价值场景建议把业务、背包和日志放进同一事务。
- `opID`、`reason`、`context` 三个字段建议业务调用时填写，方便后续检索。

## 12. 错误规则

### 12.1 常见失败原因

- 以下原因不是“任何操作都会失败”，而是“在对应操作尝试并入/合并时会失败”：

- `ownerDBID` 无效
- 数量非法
- 目标物品不存在
- 容量不足
- 物品不可堆叠
- `itemID/extra` 不一致
- 合并后超过 `maxStack`
- 交易目标背包容量不足

### 12.2 错误处理原则

- 服务端先拦截非法参数。
- 参数错误直接返回失败。
- 数据库错误写 `ERROR`。
- 客户端只看结果，不参与规则判断。

## 13. 推荐业务口径

建议在项目里统一这些口径：

- 奖励、邮件、商城、GM 都走 `opID`。
- `locked` 物品默认不允许交易。
- 限时物品过期后由业务层决定是否自动清理。
- 交易前先做目标容量预检。
- 一切整理类操作都以 `bagIndex` 为最终展示依据。

## 14. 最小决策流程

当服务端要处理一个背包动作时，建议按这个顺序判断：

1. 玩家是否有效。
2. 目标物品是否存在。
3. 数量是否合法。
4. 是否触发容量限制。
5. 物品是否允许堆叠。
6. `itemID`、`extra`、`bindType`、`locked` 是否满足规则。
7. 是否需要写日志。
8. 是否需要通知客户端。

## 15. 后续可扩展点

这份插件现在已经预留了这些方向：

- 事务版跨玩家交易
- 绑定规则细分
- 限时道具到期处理
- 锁定策略细分
- 重量/体积型容量
- 背包分页缓存
- 批量变更回调

## 16. 一句话总结

Bag 的核心规则可以概括成一句话：

> 服务端决定能不能改，数据库负责存结果，客户端只负责看和发整理请求。
